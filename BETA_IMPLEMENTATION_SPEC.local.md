# Beta Implementation Spec

## Статус документа

Цей файл є локальною інструкцією до реалізації.
Його не потрібно комітити в репозиторій.

## Головна мета

Потрібно підготувати публічно доступний бета-прототип, який можна надіслати мамі та іншим родичам як звичайний робочий сайт.

Для користувача сайт має виглядати як справжній продукт, а не як технічне демо:

- можна знайти людину;
- можна відкрити її картку;
- можна перейти до предків, нащадків, своєї гілки та всього дерева;
- можна редагувати дані;
- можна додавати нових людей;
- усе має працювати стабільно на телефоні;
- після оновлення сторінки зміни мають зберігатися.

На цьому етапі допустимо, що збереження працює лише локально в браузері користувача.
Це внутрішнє технічне обмеження, але зовні сайт має сприйматися як повністю робочий.

## Продуктовий принцип

Це не граф-редактор.
Це простий сімейний застосунок для звичайних людей, насамперед для користування з телефона.

Пріоритет рішень:

1. Зрозуміло для мами та інших родичів.
2. Зручно з телефона.
3. Виглядає як живий продукт.
4. Лише після цього технічна витонченість.

## Бета-версія: що саме будуємо

### Формат релізу

- публічний сайт;
- доступний за глобальним URL;
- деплой на `Vercel`;
- без реального backend на першій ітерації;
- без реального спільного редагування між пристроями;
- зі збереженням у `localStorage`.

### Що має відчувати користувач

Користувач відкриває посилання і бачить не "схему", а людину та прості дії навколо неї.

Правильний сценарій:

1. Відкрити сайт з телефону.
2. Знайти людину через пошук.
3. Відкрити її картку.
4. Натиснути одну з простих дій:
   `Моя ветка`, `Предки`, `Потомки`, `Все дерево`.
5. За потреби перейти до карти.
6. За потреби відкрити редагування і змінити дані.

## Рішення по стеку

### Поточний бета-стек

- `Next.js`
- `TypeScript`
- `Tailwind CSS`
- `reactflow`
- локальний `JSON`
- `localStorage`
- деплой на `Vercel`

### Чому саме так

- це найпростіший шлях швидко віддати живий прототип на реальний тест;
- поточний код уже побудований на цьому стеку;
- не потрібно зараз ускладнювати проєкт базою даних лише заради першого сімейного фідбеку;
- Vercel дає найпростіший публічний запуск для Next.js.

## Важлива архітектурна вимога

Поточна бета не повинна бути захардкоджена під `JSON`.

UI не повинен залежати від конкретного джерела даних.
Потрібно одразу зробити архітектуру з майбутньою міграцією на `Supabase`.

### Базовий принцип

- інтерфейс працює з доменною моделлю;
- джерело даних приховане за repository-шаром;
- зараз використовується `JSON + localStorage`;
- пізніше це замінюється на `Supabase` без великої переробки UI.

## Архітектурна схема

### 1. Domain layer

Потрібен окремий доменний шар із нормальними типами.

Мінімально:

- `Person`
- `Union`
- `ParentChildRelation`
- `TreeSnapshot` або аналогічний агрегат для читання дерева

### 2. Repository layer

Потрібен єдиний інтерфейс доступу до даних, наприклад:

- `loadTree()`
- `saveTree()`
- `resetTree()`

### 3. Реалізації repository

На бета-етапі:

- `JsonSeedLoader` або аналог для початкових даних;
- `LocalStorageRepository` для читання та збереження локальних змін.

У майбутньому:

- `SupabaseRepository`.

### 4. Mapping layer

Потрібні окремі мапери:

- `JSON -> domain`
- `domain -> localStorage payload`
- пізніше: `Supabase -> domain`
- пізніше: `domain -> Supabase payload`

## Модель даних для бети

### Що є зараз у сирому JSON

Зараз вузол фактично має лише:

- `id`
- `label`
- `info`

А зв'язок має:

- `source`
- `target`
- `type`

Цього достатньо для чорнового старту, але цього мало як для живого продукту.

### Яка мінімальна модель потрібна для бети

Для `Person`:

- `id`
- `name`
- `yearsText`
- `shortDescription`
- `photoUrl`
- `isDraft`

Пояснення:

- `name`: ім'я людини або коротка назва вузла;
- `yearsText`: один людський рядок, наприклад `1899-1980` або `прибл. 1910-1975`;
- `shortDescription`: короткий необов'язковий опис;
- `photoUrl`: опціональне поле, поки без обов'язкового використання;
- `isDraft`: явна ознака чернетки або неповних даних.

Для `Union`:

- `id`
- `partnerIds`

Для `ParentChildRelation`:

- `id`
- `unionId`
- `childId`

## Принцип для зв'язків

У майбутній моделі дитина повинна належати не одному "батьківському вузлу", а сімейному союзу.

На бета-етапі не обов'язково повністю перебудовувати весь UI під `union` у першу ж хвилину, але внутрішня архітектура повинна рухатися саме в цей бік.

Не можна будувати нову логіку так, ніби `parent-child` назавжди залишиться єдиною моделлю.

## Поведінка редагування в бета-версії

Редагування в бета-версії обов'язково має бути доступне користувачу.

### Ключова вимога

Якщо функція редагування присутня в інтерфейсі, вона має реально працювати для мами.

Тобто:

- можна змінити ім'я;
- можна змінити роки життя або приблизні роки;
- можна змінити короткий опис;
- можна додати людину;
- можна додати партнера;
- можна додати дитину;
- зміни мають зберігатися локально після перезавантаження сторінки.

### Технічна реалізація бета-редагування

- початкові дані беруться з `JSON`;
- після першого редагування формується локальна копія дерева в `localStorage`;
- надалі користувач працює зі своєю локальною копією;
- має існувати можливість скинути локальні зміни і повернутися до базового дерева.

### Важливо

Локальне збереження - це допустиме внутрішнє технічне рішення для бети.
З точки зору UX це все одно має виглядати як нормальна робоча функція сайту.

## Що повинно працювати у бета-версії

### Обов'язково

- пошук людей;
- відкриття картки людини;
- режими `Моя ветка`, `Предки`, `Потомки`, `Все дерево`;
- перегляд родичів у блоках `Родители`, `Партнеры`, `Дети`;
- карта дерева;
- навігація назад або інший простий спосіб повернення;
- локальне редагування;
- локальне додавання людей;
- збереження локальних змін;
- простий mobile-first інтерфейс.

### Бажано в цій же ітерації

- окрема кнопка відкриття карти;
- на телефоні карта як окремий fullscreen-view або близький до цього режим;
- явна кнопка `Назад`;
- кнопка `Сбросить изменения`;
- кнопка `Вернуться к исходным данным`.

## UX-вимоги для реалізації

### Інтерфейс

- mobile-first;
- великі кнопки;
- зрозумілі людські формулювання;
- мінімум технічних слів;
- карта не повинна бути єдиною точкою входу;
- картка людини важливіша за карту.

### Що не треба тягнути на перший план

- технічні налаштування layout;
- терміни `node`, `edge`, `canvas`, `graph`;
- редакторський шум;
- зайві технічні лічильники;
- усе, що виглядає як інструмент для розробника, а не для родича.

## Стан, який має виглядати "живим"

Бета-сайт повинен створювати враження реального застосунку, навіть якщо всередині ще тимчасова реалізація.

Тобто:

- зміни не повинні губитися після рефрешу;
- інтерфейс не повинен показувати, що "це лише прототип";
- редагування не повинно виглядати фейковим;
- функції, доступні в UI, повинні давати правдоподібний результат.

## Що відкладено на пізніше

На цьому етапі не є обов'язковими:

- реальна синхронізація між різними пристроями;
- спільне редагування;
- ролі доступу;
- історія змін;
- авторизація;
- завантаження фото у хмарне сховище;
- повний backend;
- реальний `Supabase`.

Усе це буде наступним етапом після валідації UX на живому сімейному тесті.

## План міграції після бети

Після того як бета буде перевірена на реальних користувачах:

1. Закріпити фінальну доменну модель.
2. Замінити `JSON/localStorage` на `Supabase`.
3. Додати реальне збереження.
4. Додати ролі доступу.
5. Додати історію змін.
6. Додати фото та інші розширені поля.

## Критерій успіху цієї ітерації

Ітерація вважається успішною, якщо:

- сайт відкривається по публічному посиланню;
- мама може користуватися ним з телефона;
- мама може знайти людину, перейти по гілці й відредагувати дані;
- зміни залишаються після оновлення сторінки;
- сам сайт виглядає як справжній продукт, а не як технічний прототип.

## Робоче рішення для реалізації

Отже, реалізуємо так:

- публічний бета-сайт на `Vercel`;
- поточний фронтенд-стек зберігаємо;
- додаємо repository-шар;
- відриваємо UI від сирого JSON;
- додаємо локальне збереження через `localStorage`;
- залишаємо архітектуру готовою до переходу на `Supabase`.
