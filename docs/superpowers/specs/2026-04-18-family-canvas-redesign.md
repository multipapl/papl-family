# Family Canvas — redesign spec

**Date:** 2026-04-18
**Author:** brainstorm session
**Status:** approved (pending review loop)

## Context

`q:\Blender_Python\Family_tree` — приватне сімейне дерево (~150 людей) у вигляді Next.js-застосунку. Мета — зручний доступ для літніх родичів (аудиторія: бабусі/дідусі, мала компʼютерна грамотність, телефон).

### Ускладнюючий контекст

- Значна частина родичів живе в окупованих регіонах або в рф; інтернет там нестабільний, зовнішні сервіси можуть блокуватись.
- Тому готові рішення (MyHeritage, Ancestry, Geni) **не підходять**: вони вимагають стабільного підключення до зовнішніх хостів і їхніх акаунтів.
- Окремий бекенд/Supabase також відкинуто на поточну ітерацію.

### Поточний стан коду

- `FamilyTreeCanvas.tsx` — єдиний робочий екран: одне велике SVG-полотно з усіма ~150 вузлами. Нечитабельне на телефоні.
- `reactflow` у `package.json`, але фактично **не використовується**.
- `CustomNode.tsx`, `TreeMapCanvas.tsx`, `utils/layout.ts` — мертвий код.
- Попередня спроба (commit `bb2dc58`) мала 4 режими перегляду + reactflow + картку — заплутала користувачів і була викинута.
- Доменна модель (`Person`, `Union`, `ParentChildRelation`), команди редагування (`updatePerson`, `createPartnerForPerson`, `createChildForPerson`), індекси родини (`buildTreeIndexes`, `getImmediateFamily`), та локальне сховище (`browserFamilyTreeRepository`, `localStorageRepository`, `jsonToDomain`) — **вже написані й робочі**, зберігаємо.
- Дані (`src/data/familyTree.json`) — валідні, ~150 людей.

## Product concept: Родинна книжка

Одна сторінка за URL. Відкрив — бачиш одну людину в центрі. Навколо — маленька візуалізація трьох поколінь і кнопки-імена родичів. Тап по родичу — він стає центром. Назад — повертаєшся. Усе.

Єдиний режим. Без перемикачів, без «погляду зверху», без pan/zoom.

### Принципи

1. **Зрозуміло літньому з першого разу.** Без технічного жаргону («вузол», «граф», «режим» — заборонено).
2. **Одна сторінка — одна людина.** Немає «дашборду», «навігатора», сайдбарів.
3. **Великі тап-таргети** (≥48 px), прості форми.
4. **Offline після завантаження.** Редагування зберігається в localStorage; інтернет для самого застосунку не потрібен після першого відкриття (статичний бандл).
5. **Родичі — переглядачі.** Редагування існує, але не в очі: олівець у куті. Основного «хранителя» даних — користувач проекту.

## Main screen: `PersonScreen`

Структура зверху вниз, mobile-first (перевірено для 360×740 і ширше):

1. **Шапка** (sticky):
   - `← Назад` зліва (приховано, якщо історія порожня).
   - Поточне імʼя по центру (обрізається).
   - `✎` (редагувати) справа.
2. **Mini-graph** (~35–40% висоти viewport, максимум ~320 px):
   - SVG-візуалізація трьох поколінь навколо фокуса:
     - ряд 1: батьки фокуса (0–2 вузли).
     - ряд 2: фокус + партнери + брати/сестри (горизонтальний скрол всередині ряду, якщо багато).
     - ряд 3: діти фокуса.
   - Ребра: прямі лінії від батьків до фокуса/братів; горизонтальна між фокусом і партнером (пунктир); лінії від фокуса до дітей.
   - Без pan/zoom. Якщо по ширині не влазить — горизонтальний скрол одного ряду, але кадрування завжди показує фокус.
   - Тап по будь-якому вузлу → плавний перефокус + оновлення картки + додавання в історію.
   - Активний (фокус) вузол візуально виділений (обведення, збільшений розмір).
3. **Картка людини**:
   - Велика ініціалка/аватар.
   - Імʼя (великим), роки (сірим), короткий опис (1 абзац).
4. **Блоки родичів** (дубль графа текстом — доступність і зрозумілість):
   - `Батьки` · `Партнер(ка)` · `Брати й сестри` · `Діти`.
   - Кожний родич — окрема велика кнопка з імʼям (і роками дрібним шрифтом, якщо є).
   - Порожній блок показує «—» або приховано (визначиться під час реалізації).
5. **Плаваюча кнопка пошуку** (знизу справа): `🔍` → відкриває overlay.

## Search overlay

- Повноекранний оверлей.
- Поле вводу + очищення.
- Список людей, що підходять (пошук за `name + yearsText + shortDescription`, case-insensitive, нормалізація пробілів).
- Тап на результат → overlay закривається, людина стає фокусом.
- Кнопка `Закрити` або тап поза полем.

## Editing

- Тап `✎` перемикає картку в edit-mode (весь екран переходить у той же режим).
- Картка:
  - Інлайн-поля для `name`, `yearsText`, `shortDescription`.
  - Кнопки `Зберегти` / `Скасувати` внизу.
- Блоки родичів у edit-mode:
  - У кінці кожного блоку зʼявляється `＋ Додати батька/партнера/дитину`.
  - Кожна кнопка-родич має маленький `✕` справа → підтвердження: «Прибрати N з дерева?» → `Прибрати` / `Скасувати`.
- Додавання:
  - Модалка з полями `name` (обовʼязкове), `yearsText`, `shortDescription`.
  - Використовуються наявні команди `createPartnerForPerson`, `createChildForPerson`.
  - Для батьків потрібна нова команда (див. нижче).
- Збереження: після будь-якої зміни — негайний запис у localStorage.

### Видалення (deletePerson — нова команда)

Потрібно у `src/domain/familyTreeCommands.ts`:

```ts
function deletePerson(snapshot: TreeSnapshot, personId: string): TreeSnapshot
```

Поведінка:
- Видаляє людину з `snapshot.people`.
- Видаляє всі `parentChildRelations`, де `childId === personId`.
- Для `unions`, що містять `personId` у `partnerIds`:
  - Видаляє `personId` зі списку партнерів.
  - Якщо в `union` лишається 0 партнерів — видаляє `union` і всі `parentChildRelations` з `unionId === union.id` (діти того союзу повʼязуються з невідомими батьками; їхні записи залишаються, але звʼязок parent-child обривається).
  - Якщо лишається 1 партнер — `union` зберігається, але стає «самотнім» (валідна ситуація в моделі).

### Додавання батька (нова команда)

`createParentForPerson`: створює нову людину як батька вибраної. Якщо у дитини ще немає union з батьками — створюємо новий союз. Якщо вже є union з одним відомим батьком — додаємо нового як другого партнера в той самий союз.

## Data & storage

- Seed: існуючий `src/data/familyTree.json`. Перевикористовуємо `mappers/jsonToDomain.ts` і `repositories/jsonSeedLoader.ts`.
- Персистентність: існуючий `localStorageRepository` + `browserFamilyTreeRepository`. Після будь-якої мутації — запис у localStorage.
- **Імпорт/Експорт JSON** (нова функція):
  - Схована «Налаштування» (маленька шестерінка у футері або в меню під олівцем).
  - `Експортувати JSON` — скачує файл `family.json` у TreeSnapshot-форматі.
  - `Імпортувати JSON` — завантажує файл, замінює snapshot у localStorage.
  - `Скинути до початкового` — видаляє localStorage і перечитує seed.

## State management

- Легкий `useReducer` + React Context (не треба Zustand для цього обсягу).
- Стан:
  - `snapshot: TreeSnapshot` — поточні дані.
  - `focusId: string` — ID людини на екрані.
  - `history: string[]` — стек попередніх фокусів (для кнопки «Назад»).
  - `mode: "view" | "edit"` — режим картки.
  - `searchOpen: boolean` — стан оверлея пошуку.
  - `dialog: null | { kind: "add-parent" | "add-partner" | "add-child" | "delete-confirm"; targetId: string }`.
- Reducer-actions: `focus(id)`, `back()`, `toggleEdit()`, `updatePerson(...)`, `addRelative(...)`, `deletePerson(id)`, `openSearch()`, `closeSearch()`, `openDialog(...)`, `closeDialog()`, `importSnapshot(data)`, `resetSnapshot()`.

## URL

- Одна сторінка `/`. Без роутинга по людях (для простоти цієї ітерації).
- Якщо пізніше треба буде ділитися посиланням на конкретну людину — додамо `?p=<id>`.

## Build / deploy

- `next.config.ts`: `output: "export"` — статична збірка.
- Хостинг — Vercel (існуюча інтеграція вже в README згадана).

## Non-goals for this iteration

- Повна карта усього дерева з pan/zoom.
- Фотографії (поле в моделі є, UI додамо пізніше).
- Авторизація, multi-user, real-time sync.
- Supabase / бекенд.
- Роутинг по людях в URL.
- PWA (service worker, install prompts) — надмірне ускладнення для аудиторії.

## Acceptance criteria

1. Відкрив сайт → бачу Тихона, над карткою mini-graph з батьками/партнером/дітьми.
2. Тап по імені родича (в графі або в блоці кнопок) → перефокусування без видимої затримки.
3. `← Назад` повертає на попереднього родича; приховується, коли історія порожня.
4. `✎` переводить картку в редагування; зміна імені/років/опису → `Зберегти` → після `F5` зміна на місці.
5. `＋ Додати дитину` → модалка → імʼя → `Зберегти` → новий вузол зʼявляється у ряді дітей.
6. `✕` біля родича → підтвердження → видалення; звʼязки почищені, сирот-unions немає.
7. Пошук «Роза» → тап на результат → Роза стає фокусом.
8. У UI немає слів `вузол`, `граф`, `режим`, `edge`, `node`.
9. На екрані 360×740 усе читабельно без горизонтального скролу загалом (крім одного ряду mini-graph).
10. `npm run build && npm run start` працює локально; `next export` видає статичний `out/`.

## Risks & open questions

- **Ширина mini-graph:** якщо у фокуса багато братів/сестер (>5) або дітей (>5), ряд не влазить. План: горизонтальний скрол тільки того ряду, з fade-edge. Перевірити на `Іван Дорофеев` (5 дітей).
- **Кругові звʼязки:** у моделі можливі цикли через партнерство. `buildTreeIndexes` вже має захист (`visited`). Подвійна перевірка в mini-graph layout.
- **Продуктивність:** ~150 людей — не проблема, повний індекс перераховується <10 ms. Не оптимізуємо передчасно.
- **localStorage quota:** 5 MB. Наш snapshot <50 KB. Безпечно.
