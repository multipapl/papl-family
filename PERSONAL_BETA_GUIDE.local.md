# Personal Beta Guide

## Що це за файл

Це проста особиста інструкція для мене:
що саме я маю зробити руками, щоб запустити бета-версію сайту для мами.

Цей файл локальний.
Його не треба комітити.

## Найпростіше пояснення

- `GitHub` = місце, де лежить код
- `Vercel` = місце, де сайт живе в інтернеті
- `Supabase` = база даних і бекенд, але не зараз

На поточному етапі мені потрібні:

- `GitHub`
- `Vercel`

`Supabase` поки не потрібен для першого живого тесту.

## Що ми робимо прямо зараз

Мета:

- викласти сайт в інтернет;
- відкрити його з телефона;
- скинути посилання мамі;
- зібрати фідбек.

На цьому етапі сайт буде:

- публічний;
- живий;
- схожий на справжній продукт;
- але з локальним збереженням у браузері, а не в базі.

## Що я маю зробити зараз

### Крок 1. Переконатися, що код у Git

Якщо репозиторій уже є, нічого нового тут не потрібно.

Що важливо:

- код має бути в локальному git-репозиторії;
- бажано мати гілку `main`;
- бажано, щоб проєкт був залитий на GitHub.

## Крок 2. Якщо GitHub-репозиторію ще нема, створити його

Що треба зробити:

1. Зайти в GitHub.
2. Створити новий репозиторій.
3. Назвати його якось зрозуміло, наприклад `family-tree` або `family-canvas`.
4. Залити туди поточний код.

Якщо дуже по-простому, то логіка така:

1. Створив репозиторій на GitHub.
2. Прив'язав локальний проєкт до нього.
3. Запушив код.

## Крок 3. Створити акаунт на Vercel

Що треба зробити:

1. Зайти на `vercel.com`.
2. Увійти через GitHub.
3. Дати Vercel доступ до потрібного репозиторію.

Найпростіший варіант:

- логін через GitHub;
- не вигадувати окремий вхід;
- одразу зв'язати Vercel з GitHub-репою.

## Крок 4. Задеплоїти сайт на Vercel

Що треба зробити:

1. У Vercel натиснути `New Project`.
2. Вибрати GitHub-репозиторій з цим проєктом.
3. Натиснути `Deploy`.

Що важливо:

- для Next.js Vercel зазвичай сам усе визначає правильно;
- окремо складно налаштовувати нічого не треба;
- після деплою Vercel дасть готове посилання на сайт.

## Крок 5. Відкрити сайт з телефона

Що треба перевірити руками:

1. Сайт відкривається.
2. Пошук працює.
3. Картка людини відкривається.
4. Можна перейти по родичах.
5. Можна відкрити карту.
6. Можна відредагувати дані.
7. Після оновлення сторінки зміни не зникають.

Якщо всі 7 пунктів ок, значить бета вже придатна, щоб кидати мамі.

## Крок 6. Скинути посилання мамі

Коли сайт уже працює:

1. Взяти production URL з Vercel.
2. Скинути його мамі.
3. Попросити її пройти базовий сценарій:
   - знайти людину;
   - відкрити картку;
   - перейти по гілці;
   - спробувати щось відредагувати;
   - сказати, що незрозуміло або незручно.

## Що я не маю робити зараз

Ось це поки НЕ треба:

- піднімати Supabase;
- платити за Supabase;
- писати складний backend;
- робити авторизацію;
- робити спільне редагування;
- робити ролі доступу;
- морочитися з серверною синхронізацією.

Причина проста:

спочатку треба перевірити, чи мамі взагалі зручно цим користуватися.

## Що означає "локальне збереження"

Простими словами:

- мама відкрила сайт;
- відредагувала дані;
- зміни збереглися в її браузері;
- після рефрешу вони не зникли.

Але:

- якщо вона відкриє сайт на іншому телефоні, цих змін там не буде;
- якщо інша людина відкриє те саме посилання, вона побачить свою версію або базову версію;
- це нормально для бета-етапу.

## Коли мені знадобиться Supabase

Supabase знадобиться тільки тоді, коли я скажу:

"Ок, UX уже більш-менш вірний, тепер хочу справжнє збереження."

Тобто Supabase потрібен, якщо я хочу:

- щоб зміни бачили всі;
- щоб дані жили не тільки в одному браузері;
- щоб були акаунти;
- щоб були ролі;
- щоб можна було зберігати фото;
- щоб була історія змін.

## Що я маю зробити пізніше, коли дійдемо до Supabase

### Крок 1. Створити акаунт

1. Зайти на `supabase.com`.
2. Створити акаунт.
3. Створити новий project.

### Крок 2. Зберегти головні дані про проєкт

Мені потім знадобляться:

- `Project URL`
- `anon key`

Це буде використано в застосунку.

### Крок 3. Додати ці змінні у Vercel

Коли ми дійдемо до Supabase, треба буде в `Vercel Project Settings` додати environment variables.

Швидше за все це будуть:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

Але це робимо тільки тоді, коли вже реально мігруємо на базу.

## Як мені мислити про це без технічного перевантаження

Дуже проста модель:

### Етап 1. Просто показати живий сайт

Мій стек:

- GitHub
- Vercel
- JSON
- localStorage

### Етап 2. Зробити справжнє збереження

Додається:

- Supabase

## Моє рішення без зайвих вагань

Якщо я хочу не загрузнути, то роблю так:

1. Доводжу фронт до нормальної бети.
2. Пушу код на GitHub.
3. Деплою на Vercel.
4. Кидаю мамі посилання.
5. Збираю фідбек.
6. Лише потім думаю про Supabase.

## Чекліст "готово до мами"

Перед відправкою посилання я маю перевірити:

- сайт відкривається без помилок;
- нормально працює з телефона;
- не виглядає як технічна адмінка;
- пошук простий;
- картка зрозуміла;
- редагування справді працює;
- локальні зміни не зникають;
- сайт не соромно показати людині, яка не шарить в техніці.

## Якщо зовсім коротко

Що я роблю зараз:

- `GitHub` -> покласти код
- `Vercel` -> викласти сайт
- `Supabase` -> поки не чіпати

Що я отримую:

- живий сайт за посиланням
- мама може його реально поклацати
- ми збираємо фідбек
- потім уже доробляємо справжній backend
