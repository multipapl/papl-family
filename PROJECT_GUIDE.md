# Family Canvas

Главный файл проекта для продолжения работы в новом чате.

По состоянию на 5 апреля 2026 это основной источник правды по:

- продуктовому видению;
- текущему этапу;
- честному состоянию кода;
- ближайшей бета-итерации;
- архитектуре с будущей миграцией на `Supabase`.

Если другие документы спорят с этим файлом, побеждает этот файл.

## Главный неизменный приоритет

Самая важная мысль проекта:

- приложение должно быть понятным пожилым людям;
- приложение должно быть удобным на телефоне и планшете;
- приложение не должно требовать “понимать компьютер”;
- десктоп для нас вторичен;
- красота важна, но только после простоты и понятности.

Правильный порядок всегда такой:

1. понятно пожилому родственнику;
2. удобно с телефона;
3. легко пользоваться без объяснений;
4. только потом красиво, эффектно и технологично.

## Что мы строим

Мы строим не абстрактный граф-редактор, а понятное семейное приложение с картой родства.

Базовая идея:

- пользователь сначала видит человека, а не огромную схему;
- он двигается простыми действиями: `Моя ветка`, `Предки`, `Потомки`, `Все дерево`;
- карта существует, но как второй уровень интерфейса;
- редактирование существует, но как отдельный режим, а не как постоянный шум на первом экране.

## Для кого продукт

- в первую очередь для пожилых родственников;
- в первую очередь для телефона и планшета;
- для семьи, а не для технических пользователей;
- для людей, которым нужно быстро понять “кто кому кем приходится”;
- для 1-2 более уверенных пользователей, которые будут чаще вносить изменения.

## Текущий этап проекта

Сейчас проект находится не на этапе “технического демо”, а на этапе подготовки первой публичной беты.

Цель этой беты:

- выложить сайт в интернет;
- отправить его маме и другим родственникам;
- получить живой фидбек;
- проверить, удобно ли пользоваться сайтом с телефона;
- понять, что непонятно, страшно или мешает.

Важно:

сайт должен выглядеть как настоящий рабочий продукт, даже если часть реализации пока временная.

## Что означает “публичная бета” в нашем случае

На этом этапе мы сознательно делаем не полный production с backend, а правдоподобную рабочую бету.

Пользователь должен чувствовать, что:

- сайт живой;
- поиск работает;
- навигация работает;
- редактирование работает;
- изменения не пропадают после обновления страницы.

Внутренне это может быть устроено временно.
Внешне это должно выглядеть как настоящий продукт.

## Практическое решение для беты

### Что делаем сейчас

- деплой фронтенда на `Vercel`;
- данные стартуют из локального `JSON`;
- пользовательские изменения сохраняются локально в браузере через `localStorage`;
- редактирование остается доступным;
- сайт работает по публичному URL.

### Что сознательно не делаем прямо сейчас

- настоящий backend;
- настоящую синхронизацию между устройствами;
- авторизацию;
- роли доступа;
- совместное редактирование;
- серверное хранение фото;
- реальную базу данных.

Причина:

сначала нужно проверить сам продуктовый UX на живых родственниках.

## Главные UX-принципы

### 1. Mobile first

Это в первую очередь телефонный продукт.

Это означает:

- интерфейс проектируется сначала под узкий экран;
- важный контент идет вертикально сверху вниз;
- основные действия крупные и понятные;
- карту можно двигать одним пальцем и масштабировать щипком;
- пользователь не должен “понимать layout”, чтобы пользоваться сайтом.

### 2. Card first, map second

Пользователь сначала видит карточку человека.

Правильный сценарий:

1. найти человека;
2. открыть карточку;
3. посмотреть родителей, партнеров и детей;
4. при желании перейти к карте;
5. только потом открыть все дерево.

### 3. Простые формулировки

Нельзя заставлять пользователя понимать термины:

- `layout`
- `canvas`
- `node`
- `edge`
- `focus depth`

В интерфейсе должны жить человеческие действия:

- `Моя ветка`
- `Предки`
- `Потомки`
- `Все дерево`
- `Показать больше`
- `Найти человека`
- `Редактировать`

### 4. Интерфейс должен переживать человека, который “не дружит с техникой”

Нужно исходить из того, что пользователь:

- боится что-то сломать;
- не любит сложные экраны;
- не будет читать длинные инструкции;
- хочет нажать 1-2 понятные кнопки и сразу увидеть результат.

Отсюда вывод:

- прятать сложность;
- не перегружать первый экран;
- не показывать технические настройки без необходимости;
- опираться на карточки, крупные кнопки и простые формулировки.

### 5. Поиск важнее ручной навигации

На большом дереве поиск по имени должен быть самым быстрым способом попасть в нужную точку.

## Честное состояние текущего кода

### Что уже есть в кодовой базе

Текущий код уже умеет:

- показывать mobile-first экран;
- открывать карточку выбранного человека;
- искать по имени и заметке;
- показывать блоки `Родители / Партнеры / Дети`;
- переключать режимы `Моя ветка / Предки / Потомки / Все дерево`;
- показывать карту дерева;
- двигать и масштабировать карту на тач-экране;
- включать локальный режим редактирования;
- добавлять ребенка или партнера;
- удалять карточку локально в рамках текущей сессии.

### Что в коде пока еще временное или неполное

На текущий момент в коде еще нет:

- сохранения изменений после перезагрузки страницы;
- repository-слоя между UI и данными;
- отдельной доменной модели;
- полноценной модели `person / union / relation`;
- backend;
- базы данных;
- авторизации;
- ролей доступа;
- фото;
- истории изменений.

### Важный честный факт

Сейчас редактирование уже есть, но оно пока живет только в оперативном состоянии страницы.

Следующий шаг:

- сделать так, чтобы оно сохранялось локально через `localStorage`;
- при этом для пользователя продолжало выглядеть как обычная рабочая функция сайта.

## Архитектурное направление

Главная архитектурная мысль:

UI не должен быть навсегда захардкожен под `JSON`.

### Правильная схема

- UI работает с доменной моделью;
- данные приходят через repository-слой;
- текущий источник данных скрыт за абстракцией;
- в бете используются `JSON + localStorage`;
- позже реализация меняется на `Supabase`, а UI остается почти тем же.

### Что это означает practically

Нужно перейти от схемы:

- `компонент напрямую читает JSON`

к схеме:

- `компонент -> repository -> domain model`

## Целевая архитектура для беты

### 1. Domain layer

Нужны нормальные предметные типы:

- `Person`
- `Union`
- `ParentChildRelation`
- `TreeSnapshot` или аналогичный агрегат для чтения дерева

### 2. Repository layer

Нужен единый интерфейс доступа к данным, например:

- `loadTree()`
- `saveTree()`
- `resetTree()`

### 3. Реализации repository

На этапе беты:

- источник seed-данных из `JSON`;
- локальное сохранение через `localStorage`.

Позже:

- `SupabaseRepository`.

### 4. Mapping layer

Нужны маперы:

- `JSON -> domain`
- `domain -> localStorage payload`
- позже: `Supabase -> domain`
- позже: `domain -> Supabase payload`

## Модель данных

### Что есть сейчас в сыром источнике

Сейчас узел фактически хранит:

- `id`
- `label`
- `info`

А связь хранит:

- `source`
- `target`
- `type`

Этого хватает для текущего прототипа, но этого мало для нормальной эволюции продукта.

### Минимальная разумная модель для беты

Для `Person`:

- `id`
- `name`
- `yearsText`
- `shortDescription`
- `photoUrl`
- `isDraft`

Пояснение:

- `name` — имя или основное отображаемое название;
- `yearsText` — человекочитаемый текст вроде `1899-1980` или `прибл. 1910-1975`;
- `shortDescription` — краткая заметка;
- `photoUrl` — опционально, пока без обязательного использования;
- `isDraft` — явная техническая отметка, а не эвристика по названию.

### Для родственных связей

Для `Union`:

- `id`
- `partnerIds`

Для `ParentChildRelation`:

- `id`
- `unionId`
- `childId`

### Принцип на будущее

Ребенок должен принадлежать не одному “родителю-узлу”, а семейному союзу.

Именно к этой модели нужно двигаться.

## Поведение редактирования в бете

Редактирование в бете должно быть доступно обычному пользователю.

Если кнопка редактирования есть в интерфейсе, она должна реально работать.

Пользователь должен иметь возможность:

- изменить имя;
- изменить годы жизни или приблизительные годы;
- изменить короткий текст;
- добавить ребенка;
- добавить партнера;
- удалить локально добавленную карточку;
- увидеть свои изменения после перезагрузки страницы.

### Внутреннее техническое решение

- стартовые данные загружаются из `JSON`;
- после изменений пользователь работает со своей локальной копией в `localStorage`;
- должна существовать возможность сбросить локальные изменения и вернуться к исходному дереву.

## Как должен использоваться продукт

### Базовый семейный сценарий

1. Человек открывает ссылку с телефона.
2. Видит поиск и выбранную карточку человека.
3. Находит нужного родственника.
4. Нажимает `Предки`, чтобы понять линию выше.
5. Нажимает `Потомки`, чтобы увидеть детей и внуков.
6. При желании открывает `Все дерево`.

### Сценарий уточнения данных

1. Родственник находит человека.
2. Открывает карточку.
3. Включает режим редактирования.
4. Исправляет имя, годы или заметку.
5. При необходимости добавляет ребенка или партнера.
6. Обновляет страницу и видит, что изменения не пропали.

### Сценарий для пожилого пользователя

1. Открывает ссылку.
2. Нажимает на поиск.
3. Вводит имя.
4. Нажимает на найденного человека.
5. Пользуется большими понятными кнопками без чтения инструкций.
