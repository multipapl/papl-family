# Family Canvas

Главный файл проекта для продолжения работы в новом чате.

## Главный неизменный приоритет

Самая важная мысль проекта:

- это приложение должно быть понятным пожилым людям;
- это приложение должно быть удобным на телефоне и планшете;
- это приложение не должно требовать “понимать компьютер”;
- десктоп для нас вторичен;
- красота важна, но только после простоты и понятности.

Если в будущем какая-то идея спорит с этим приоритетом, приоритет побеждает.

Правильный порядок такой:

1. понятно пожилому родственнику;
2. удобно с телефона;
3. легко пользоваться без объяснений;
4. только потом красиво, эффектно и технологично.

## Что мы строим

Не просто “редактор графа”, а понятное семейное приложение с картой родства.

Основная идея:

- обычный родственник открывает не огромную схему, а карточку человека;
- дальше он двигается простыми действиями: `Предки`, `Потомки`, `Моя ветка`, `Все дерево`;
- карта существует, но как второй уровень интерфейса, а не как единственная точка входа;
- редактирование должно быть отдельным и спокойным режимом, а не постоянным шумом на экране.

## Для кого продукт

- в первую очередь для пожилых родственников;
- в первую очередь для телефона и планшета;
- для семьи, а не для технических пользователей;
- для людей, которые чаще открывают сайт с телефона, чем с компьютера;
- для родственников, которым нужно быстро понять “кто кому кем приходится”, а не разбираться в канвасе;
- для 1-2 более уверенных пользователей, которые будут вносить основную часть изменений.

## Главные UX-принципы

### 1. Mobile first

Это в первую очередь телефонный продукт.

Что это означает:

- десктопный UX не должен диктовать структуру интерфейса;
- интерфейс проектируется сначала под узкий экран;
- все основные кнопки крупные, нажатие пальцем должно быть комфортным;
- карта должна нормально двигаться одним пальцем и масштабироваться щипком;
- важный контент должен идти вертикально сверху вниз, без необходимости “понимать layout”.

### 2. Card first, map second

Пользователь сначала видит человека, а не огромную сеть.

Правильный сценарий:

1. найти человека;
2. открыть карточку;
3. посмотреть родителей / партнеров / детей;
4. при желании перейти к карте;
5. только потом открывать все дерево.

### 3. Простые формулировки

Нельзя заставлять пользователя понимать термины вроде:

- `focus depth`
- `layout`
- `canvas`
- `node`
- `edge`

В интерфейсе должны жить человеческие действия:

- `Моя ветка`
- `Предки`
- `Потомки`
- `Все дерево`
- `Показать больше`
- `Найти человека`
- `Редактировать`

### 3.1. Интерфейс должен переживать пользователя, который “не очень дружит с техникой”

Это очень важный фильтр для всех следующих решений.

Нужно исходить из того, что человек:

- может бояться “что-то сломать”;
- не любит сложные экраны;
- не понимает разницу между режимами, если ее нужно объяснять;
- не будет читать длинные инструкции;
- хочет нажать 1-2 понятные кнопки и сразу увидеть результат.

Отсюда вывод:

- прятать сложность;
- не перегружать первый экран;
- не показывать редакторские настройки без необходимости;
- чаще использовать карточки и большие кнопки, чем панели инструментов.

### 4. Редактирование скрыто по умолчанию

Обычный пользователь должен видеть просмотр.

Редактирование открывается отдельно и не должно перегружать первый экран.

### 5. Поиск важнее ручной навигации

На большом дереве поиск по имени должен быть самым быстрым способом попасть в нужную точку.

## Что уже сделано в текущем локальном прототипе

Текущий прототип без базы данных и без авторизации уже проверяет главную идею:

- экран стал mobile-first;
- интерфейс переведен на русский;
- первый акцент теперь на карточке человека;
- карта вынесена в отдельную большую секцию;
- есть понятные режимы `Моя ветка / Предки / Потомки / Все дерево`;
- есть быстрый поиск;
- есть блоки `Родители / Партнеры / Дети`;
- есть локальный режим редактирования, скрытый по умолчанию;
- карту можно двигать и масштабировать на тач-экране.

## Что все еще нужно докручивать

### UX

- стартовый экран должен быть еще проще;
- возможно, стоит сделать старт не с Тихона, а с “точки входа семьи”;
- нужно добавить явную кнопку `Назад`;
- стоит сделать отдельный режим `Моя линия` или `Мой род`;
- возможно, карту на телефоне стоит открывать в полноэкранном режиме по кнопке.

### Навигация

- нужен “хлебный путь” более понятный для семьи;
- нужен режим “показать только ближайших родственников”;
- нужен режим “показать ветку целиком” без лишнего шума соседних линий;
- нужно продумать быстрый возврат из глубокой навигации.

### Контент

- позже нужна карточка человека с фото, датами, местами, заметками и документами;
- карта не должна быть единственным способом чтения семейной информации.

## Как я вижу использование продукта

### Базовый семейный сценарий

1. Человек открывает ссылку с телефона.
2. Видит поиск и выбранную карточку человека.
3. Нажимает `Предки`, чтобы понять линию выше.
4. Нажимает `Потомки`, чтобы увидеть детей и внуков.
5. Если хочет общий контекст, открывает `Все дерево`.

### Сценарий уточнения данных

1. Родственник находит человека.
2. Открывает карточку.
3. Включает `Редактирование`.
4. Исправляет имя/заметку или добавляет ребенка/партнера.
5. Позже это будет сохраняться в базу данных.

### Сценарий для пожилого пользователя

1. Открывает ссылку.
2. Нажимает на поиск.
3. Вводит имя.
4. Нажимает на найденного человека.
5. Дальше нажимает только крупные понятные кнопки.

## Текущий стек

По состоянию на 5 апреля 2026:

- `Next.js 16`
- `React 19`
- `TypeScript`
- `Tailwind CSS 4`
- `reactflow`
- `dagre`
- локальный `JSON` как источник данных

### Что в текущем стеке хорошо

- быстро стартует;
- удобно собирать локальные прототипы;
- легко выкатывать UI-итерации;
- хватает для проверки UX без сервера.

### Что в текущем стеке временное

- `familyTree.json` как хранилище данных;
- `dagre` как layout-движок;
- отсутствие отдельного слоя доменной модели;
- отсутствие базы данных и авторизации.

## Будущий стек: рекомендованный

### Вариант A. Самый практичный

- `Next.js`
- `React`
- `TypeScript`
- `Tailwind CSS`
- `@xyflow/react`
- `elkjs`
- `Zustand`
- `Zod`
- `Supabase`
- `Postgres`
- `Storage`
- `Auth`
- `Realtime`
- деплой либо на `Vercel`, либо на `Cloudflare`

### Почему я сейчас рекомендую именно этот вариант

- он самый сбалансированный по времени разработки;
- есть SQL-база, что очень хорошо подходит под родственные связи;
- `Supabase` закрывает сразу базу, auth, storage и realtime;
- это дешевле и проще, чем собирать 4 сервиса вручную;
- с ним легче прийти к совместному редактированию без сильной инфраструктурной боли.

### Что я бы поменял относительно текущего прототипа

- `reactflow` -> `@xyflow/react`
- `dagre` -> `elkjs`
- `JSON` -> `Postgres`
- локальный state -> `Zustand`
- простая структура `nodes/edges` -> предметная модель `person / union / relation / note / media`

## Будущий стек: самый дешевый вариант

### Вариант B. Если минимизировать ежемесячную стоимость

- `Next.js` или `React + Vite`
- `Cloudflare Workers / Pages`
- `Cloudflare D1`
- `Cloudflare R2`
- `Auth.js` или простая собственная auth-логика

### Плюсы

- очень дешево;
- хорошие бесплатные лимиты;
- бесплатная раздача статики;
- бесплатный или почти бесплатный рантайм.

### Минусы

- сложнее в разработке;
- меньше “готового из коробки”, чем у Supabase;
- роли, авторизация, realtime и storage придется собирать более вручную.

### Мой честный вывод

Если приоритет номер один — быстро и спокойно довести продукт до живого семейного использования, я бы все равно начал с `Supabase`.

Если приоритет номер один — экономить каждый доллар и готов мириться с большей сложностью разработки, тогда уже смотреть в сторону `Cloudflare`.

## Что я рекомендую по backend и deploy

### Моя рекомендация на сегодня

Для первых реальных версий:

- `Supabase` как backend
- `Vercel Hobby` как деплой фронтенда

Это самый простой путь.

### Если захочется дешевле

Второй кандидат:

- `Cloudflare Workers / Pages + D1 + R2`

Но это уже решение “дешевле деньгами, дороже разработкой”.

### Если нужен только Postgres, но не нужен полный BaaS

Можно рассматривать:

- `Neon` для Postgres
- отдельно auth
- отдельно storage

Но в сумме это обычно менее удобно, чем `Supabase`, если нужен именно полноценный продукт, а не только база.

## Коротко по ценам и лимитам

Ниже не “навсегда”, а ориентир по официальным страницам на 5 апреля 2026.

### Supabase

- Free: 2 free projects
- Free: `500 MB` database size per project
- Free: `50,000 MAU`
- Free: `2 million` realtime messages
- Pro: `25 USD / month`
- Pro: includes `10 USD` compute credits

Практический вывод:

- для прототипа и первых тестов хватит;
- когда появится живая семья и регулярные изменения, это все еще очень разумный вариант.

### Vercel

- Hobby: free forever
- Hobby: `1M` edge requests per month included
- Hobby: `100 GB` fast data transfer included
- Hobby: `1M` function invocations included

Практический вывод:

- для frontend-части и демо-стадий этого более чем достаточно.

### Cloudflare

- Workers Free: `100,000` requests per day
- Workers Paid: `5 USD / month`, includes `10 million` requests per month
- Pages static assets: free and unlimited
- D1 Free: `5 million` rows read per day
- D1 Free: `100,000` rows written per day
- D1 Free: `5 GB` storage total

Практический вывод:

- как платформа “максимально дешево” очень сильный вариант.

### Neon

- Free: `100` projects
- Free: `100 CU-hours` monthly per project
- Free: `0.5 GB` storage per project
- Free: `60K MAU` via Neon Auth
- платный Launch: usage-based, typical spend shown as `15 USD / month`

Практический вывод:

- сильный кандидат, если нужен именно Postgres;
- но по сумме продукта `Supabase` для нашего кейса все еще удобнее.

## Будущая модель доступа

Пока не реализовано, но нужно держать в голове:

- `Owner` — полный доступ
- `Editor` — может менять дерево
- `Contributor` — предлагает изменения
- `Viewer` — только смотрит

Моя предварительная мысль:

- большинству родственников лучше давать `Viewer` или `Contributor`;
- прямое редактирование стоит оставлять 1-2 людям;
- позже нужен журнал изменений и история правок.

## Какая модель данных нам нужна в будущем

Не просто `nodes/edges`.

Нужны сущности:

- `persons`
- `unions` или `family_units`
- `parent_child_relations`
- `notes`
- `media`
- `sources`
- `board_views`
- `permissions`

Главная идея:

ребенок должен принадлежать не одному “узлу-родителю”, а семейному союзу/ячейке. Это даст более правильный и читаемый family tree.
